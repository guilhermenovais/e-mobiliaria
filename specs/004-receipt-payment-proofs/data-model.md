# Data Model: Receipt Payment Proofs

## New Entity: `PaymentProof`

**Package**: `receipt/domain/entity/`

| Field              | Type            | Nullable | Constraint                                     |
|--------------------|-----------------|----------|------------------------------------------------|
| `id`               | `Long`          | No       | Auto-assigned by DB; null until persisted      |
| `originalFileName` | `String`        | No       | User-visible name; non-blank                   |
| `storedFileName`   | `String`        | No       | UUID-based stored name; non-blank              |
| `fileType`         | `ProofFileType` | No       | Enum: `PDF` or `IMAGE`                         |
| `attachedAt`       | `LocalDate`     | No       | Date the proof was attached; not null          |
| `receiptId`        | `Long`          | No       | FK to owning receipt; set at construction time |

**Factory methods** (following existing pattern):

- `PaymentProof.create(originalFileName, storedFileName, fileType, attachedAt, receiptId)` — for new proofs
- `PaymentProof.restore(id, originalFileName, storedFileName, fileType, attachedAt, receiptId)` — for DB hydration

**Validation rules**:

- `originalFileName` must not be null or blank
- `storedFileName` must not be null or blank
- `fileType` must not be null
- `attachedAt` must not be null
- `receiptId` must not be null

## New Enum: `ProofFileType`

**Package**: `receipt/domain/entity/`

| Value   | Accepted extensions                                               |
|---------|-------------------------------------------------------------------|
| `PDF`   | `.pdf`                                                            |
| `IMAGE` | `.jpg`, `.jpeg`, `.png`, `.gif`, `.bmp`, `.webp`, `.tiff`, `.tif` |

Utility method: `ProofFileType.fromExtension(String ext)` → returns `Optional<ProofFileType>` (empty if unsupported).

## Modified Entity: `Receipt`

**Added field**:

```
List<PaymentProof> proofs   // mutable list, initialized to empty ArrayList
```

**Added method**:

- `Receipt.hasProofs()` → `boolean` — returns `!proofs.isEmpty()`
- `Receipt.getProofs()` → unmodifiable view of the list

**`restore()` overload** (new, existing overload unchanged to avoid breaking existing tests):

```java
Receipt.restoreWithProofs(id, date, paymentDueDate, intervalStart, intervalEnd,
                          discount, fine, observation, contract, List<PaymentProof> proofs)
```

The existing `restore()` sets proofs to an empty list. The new overload is used by the JDBC repository when loading
proofs alongside receipts.

## Database Migration: `V10__add_payment_proofs.sql`

```sql
CREATE TABLE payment_proofs (
    id                BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    receipt_id        BIGINT       NOT NULL REFERENCES receipts(id) ON DELETE CASCADE,
    original_filename VARCHAR(255) NOT NULL,
    stored_filename   VARCHAR(255) NOT NULL UNIQUE,
    file_type         VARCHAR(10)  NOT NULL,
    attached_at       DATE         NOT NULL
);

CREATE INDEX idx_payment_proofs_receipt_id ON payment_proofs(receipt_id);
```

## AppDataPaths Change

New method added to `AppDataPaths`:

```java
public static Path proofStorageDirectory() {
    return ensureDirectory(resolveAppDataDir().resolve("proofs"));
}
```

This resolves to `%LOCALAPPDATA%\e-mobiliaria\proofs\` on Windows, `~/.e-mobiliaria/proofs/` on other OS.

## New Domain Service Interface: `PaymentProofStorageService`

**Package**: `receipt/domain/service/`

```java
public interface PaymentProofStorageService {
    // Copy a file from disk into proof storage; returns the stored filename
    String copyToStorage(Path sourceFile, String originalFileName);
    // Copy clipboard image bytes into proof storage; returns the stored filename
    String copyBytesToStorage(byte[] imageBytes, String originalFileName);
    // Delete a stored proof file by its stored filename
```

## Entity Relationships (updated)

```
Receipt  1 ──── * PaymentProof
   ^
   |
  Contract (existing, unchanged)
```

- One `Receipt` has zero or more `PaymentProof` items
- Each `PaymentProof` belongs to exactly one `Receipt`
- Deleting a `Receipt` cascades to delete its `PaymentProof` DB rows (ON DELETE CASCADE)
- The file system files are deleted by `DeleteReceiptInteractor` before the DB row is removed
