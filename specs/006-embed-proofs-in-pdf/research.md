# Phase 0 Research: Embed Payment Proofs as PDF Pages

All unknowns raised by the Technical Context have been resolved below. No `NEEDS CLARIFICATION` markers remain.

## 1. PDF page-manipulation library

- **Decision**: Use OpenPDF directly (`com.lowagie.text.pdf.PdfReader`, `PdfCopy`, `PdfWriter`, `PdfContentByte`,
  `com.lowagie.text.Image`) — group/artifact `com.github.librepdf:openpdf:1.3.30`, already a `pom.xml` dependency.
- **Rationale**: OpenPDF is already on the classpath (used transitively by `jasperreports-pdf` to export the Jasper
  report to PDF), is well-precedented for PDF work in this ecosystem, and its `PdfContentByte.addTemplate` API gives
  the low-level control needed to draw a page's content at an arbitrary scale/offset — required to implement the
  "contain and center" placement FR-004 demands. It was not previously `requires`'d directly in `module-info.java`
  because `PdfGenerationService` only calls into JasperReports, never into OpenPDF classes itself.
- **Alternatives considered**: Apache PDFBox — equally capable, but would add a second, redundant PDF engine to a
  project that already ships one; rejected purely to avoid duplicate dependencies doing the same job.

## 2. Determining the receipt's fixed page size

- **Decision**: After generating the receipt's own PDF bytes via the existing `PdfGenerationService` (unchanged),
  open them with `PdfReader` and read `reader.getPageSizeWithRotation(1)`. Use that `Rectangle` as the fixed page
  size for every page appended for proofs.
- **Rationale**: The receipt's page size (595×842pt / A4, 20pt margins on all sides, per `receipt.jrxml`) is defined
  in the Jasper template, not in Java code. Deriving it from the already-generated PDF avoids a second, hand-copied
  source of truth that would silently drift if the `.jrxml` is ever changed.
- **Alternatives considered**: Hardcode A4 dimensions in the new service — rejected for the drift risk above.

## 3. Fast path when a receipt has no proofs

- **Decision**: If `receipt.getProofs()` is empty, return the Jasper-generated bytes completely untouched — do not
  invoke OpenPDF at all.
- **Rationale**: FR-005 / SC-004 require byte-for-byte-equivalent output to current (pre-feature) behavior for
  receipts without proofs. Even a no-op round-trip through OpenPDF (read + `PdfCopy` + write) can alter incidental PDF
  structure/metadata; skipping the merge path entirely is the only way to guarantee zero regression.

## 4. Building the merged document (fixed page size + "contain" scaling)

- **Decision**: Build the output using `PdfCopy` in import mode:
    1. Import every page of the base receipt PDF unchanged (`writer.getImportedPage` + `copy.addPage`).
    2. For each proof, in attachment order, produce one or more *new* pages, each sized exactly like the receipt's page
       (from decision #2). On each new page, draw the proof's content — a decoded raster image for an image proof, or
       an imported template of one source page for a PDF proof — scaled uniformly so it fits entirely within the page
       (i.e., `scale = min(pageWidth / contentWidth, pageHeight / contentHeight)`), then centered, leaving blank margins
       where the aspect ratios differ.
- **Rationale**: This is the only construction that satisfies FR-002/FR-003 (proof content shown in full, undistorted,
  at its own aspect ratio) **and** FR-004 (every output page — including proof pages — shares the receipt's fixed
  page size) simultaneously. A PDF proof's pages are treated as opaque rendered content (via
  `PdfContentByte.addTemplate`)
  rather than parsed/re-flowed, so nothing about the source page's internal layout needs to be understood.
- **Alternatives considered**: Append each proof's PDF pages verbatim via `PdfCopy`, keeping their original page
  size — simpler, but produces a document with mixed page sizes, directly violating FR-004; rejected.

## 5. Image proof orientation (EXIF)

- **Decision**: Add `com.drewnoakes:metadata-extractor:2.19.0` as a new dependency. Before wrapping a decoded image
  in `com.lowagie.text.Image`, read its EXIF `Orientation` tag (`ExifIFD0Directory.TAG_ORIENTATION`) via
  `ImageMetadataReader.readMetadata`, and if present and non-default, rotate/flip the `BufferedImage` with an
  `AffineTransform` so the in-memory pixels match the visually-correct orientation before it is placed on the page.
- **Rationale**: OpenPDF's `Image.getInstance` does not consult EXIF orientation. `javax.imageio`'s standard metadata
  tree does not expose a portable orientation tag for JPEG across JVM vendors. `metadata-extractor` is the
  established, dependency-free (verified via its POM — no transitive runtime dependencies, single ~370KB jar) Java
  library for this exact problem, satisfying the spec's "attached image uses an orientation flag different from its
  stored pixel orientation" edge case.
- **Module system note**: The jar ships no `Automatic-Module-Name` manifest entry, so on the module path it resolves
  to the derived automatic module name `metadata.extractor` (dashes → dots, from the jar's base file name). This is
  used in the `requires metadata.extractor;` line added to `module-info.java`.
- **Alternatives considered**: Hand-parsing the JPEG APP1/Exif segment — rejected as unnecessary reimplementation of a
  solved, easy-to-get-subtly-wrong problem (endianness, IFD offsets, multiple tag formats).

## 6. Per-proof failure isolation ("all-or-nothing" per proof)

- **Decision**: Each proof is processed inside its own try/catch, independent of the others. Any exception while
  resolving the stored file (missing), decoding it (corrupt image), or reading/importing it as a PDF (corrupt,
  encrypted, or a failure partway through a multi-page PDF proof) causes that entire proof's staged pages (if any were
  already produced) to be discarded, records it in the skip list with a reason, and processing moves on to the next
  proof and finally to finishing the document.
- **Rationale**: Matches the spec's clarified answer that a partially-corrupt PDF proof is skipped in its entirety
  (no partial-page inclusion), and FR-006's requirement that one bad proof must not abort the whole generation.

## 7. Surfacing skipped proofs to the user

- **Decision**: `PaymentProofPdfEmbeddingService.embed(...)` returns a `ProofEmbeddingResult` carrying both the final
  PDF bytes and a `List<SkippedProof>` (empty when nothing was skipped). `GenerateReceiptPdfInteractor` maps that list
  into `GenerateReceiptPdfOutput.skippedProofs()` (a UI-friendly `List<SkippedProofInfo>` of display name + reason).
  `ReceiptListController.handleGeneratePdf` shows a `javafx.scene.control.Alert` (information/warning style) listing
  the omitted proofs and reasons once the background generation `Task` succeeds, but only when the list is non-empty;
  nothing changes for the common case of zero skipped proofs.
- **Rationale**: Directly implements the spec's clarified answer ("Dialog after generation — a modal/alert dialog
  appears once PDF generation finishes, listing which proof(s) were skipped and why") and reuses the `Alert` pattern
  already used elsewhere in `ReceiptListController` (e.g., the delete-confirmation dialog).

## 8. Where the embedding logic sits in the layering

- **Decision**: `PaymentProofPdfEmbeddingService` is a **domain service interface** in `receipt/domain/service/`
  (alongside `ReceiptFileService` and `PaymentProofStorageService`), with a single real implementation,
  `OpenPdfProofEmbeddingService`, in `receipt/infrastructure/service/`, plus a `FakePaymentProofPdfEmbeddingService`
  in the matching test package for interactor tests, per `docs/domain-services.md`'s rule that PDF-generation-style
  services get a fake for tests even though (unlike e.g. a web-API-backed service) a real implementation is also
  provided.
- **Rationale**: The interface depends on the `PaymentProof` domain entity and is orchestrated by
  `GenerateReceiptPdfInteractor` alongside `ReceiptFileService`, matching the existing shape of the feature. Keeping
  it a `receipt`-local domain service (rather than, say, a generic `shared/pdf` utility) avoids `shared` depending on
  the `receipt` feature's `PaymentProof` entity, which would invert the intended `shared` → `feature` dependency
  direction described in `docs/general-instructions.md`.
- **Alternatives considered**: A generic, entity-agnostic "append pages to PDF" utility in `shared/pdf` that
  `ReceiptFileServiceImpl` calls directly — rejected because it would need to either accept raw byte arrays per page
  (pushing proof-type-specific decoding logic — EXIF handling, PDF-vs-image branching — into `ReceiptFileServiceImpl`,
  bloating a class whose sole current job is "call Jasper") or take on a dependency on `PaymentProof`, which belongs
  to the `receipt` feature.
