# Implementation Plan: Receipt Payment Day Selection

**Branch**: `001-receipt-payment-day` | **Date**: 2026-05-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-receipt-payment-day/spec.md`

## Summary

Add a `paymentDueDate` field to `Receipt` that identifies which rent obligation a payment covers.
Replace the free-form interval entry on the receipt form with a structured payment day selector
showing only unreceipted due dates. Centralize due date computation in a new `PaymentDueDateService`
(pure domain logic) consumed by the receipt form, dashboard, and future reports.

## Technical Context

**Language/Version**: Java 24  
**Primary Dependencies**: JavaFX (UI), Google Guice 7 (DI), JDBC + H2 (persistence), Flyway
(migrations — currently at V7), JUnit 5 (tests)

`FakeContractRepository`, etc.)  
**Target Platform**: Desktop application (JavaFX, single user)  
**Project Type**: Desktop application  
**Performance Goals**: Interactive responsiveness; due date computation is pure in-memory (no
performance concern for the dataset size)  
**Constraints**: Clean layered architecture (domain → application → infrastructure/ui); no
framework/UI dependencies in domain or application layers  
**Scale/Scope**: Single-property-manager desktop app; typical dataset: tens to hundreds of contracts
and receipts

## Implementation Sequence

Tasks must be implemented in this dependency order (to be detailed in `tasks.md`):

1. **DB migration** (`V8__add_receipt_payment_due_date.sql`) — schema foundation for everything else
2. **`PaymentDueDateService`** — pure logic, no dependencies; needed by interactors and dashboard
3. **`Receipt` entity** — add `paymentDueDate` field + validation
4. **`ReceiptRepository` interface** — new query methods
5. **`FakeReceiptRepository`** — fake implementation of new methods (required by tests)
6. **`CreateReceiptInput` / `EditReceiptInput`** — add `paymentDueDate` field
7. **`GetUnreceiptedDueDatesInput/Output`** — new use case I/O records
8. **`GetUnreceiptedDueDatesInteractor`** — new use case
9. **`CreateReceiptInteractor`** — add `paymentDueDate` + uniqueness enforcement
10. **`EditReceiptInteractor`** — add `paymentDueDate` + uniqueness enforcement
11. **`JdbcReceiptRepository`** — include `payment_due_date` in all SQL
12. **`JdbcDashboardRepository`** — replace interval-based unpaid rent query
13. **`ReceiptFormController`** — payment day selector UI
14. **`module-info.java`** — open any new packages to Guice/FXML
15. **`ReceiptModule`** — bind `GetUnreceiptedDueDatesInteractor` if needed
16. **Tests** — `PaymentDueDateServiceTest`, updated interactor tests, new interactor test

## Key Design Decisions

All decisions and their rationale are documented in `research.md`. Summary:

| Decision                      | Choice                                                          |
|-------------------------------|-----------------------------------------------------------------|
| Due date computation location | `contract/domain/service/PaymentDueDateService` (pure class)    |
| Legacy migration algorithm    | `paymentDay` clamped to the month of `interval_start`           |
| Uniqueness enforcement        | DB UNIQUE constraint + application-level `BusinessException`    |
| Dashboard migration           | Java-side due date computation + single batch receipt query     |
| Interval pre-fill rule        | `intervalStart = D`, `intervalEnd = D + 1 month − 1 day`        |
| Edit mode selector            | Current due date + other unreceipted dates                      |
| Reports scope                 | No changes required (reports don't surface pending rents today) |
